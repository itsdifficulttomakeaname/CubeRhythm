# 判定系统问题修复 - 第二轮

## 修复日期
2025-02-14

## 问题描述

用户报告了两个判定系统问题：

1. **FLICK 判定有误**: 转头没有判定
2. **HOLD 只需要按着而不需要瞄准**: HOLD 音符应该只检测按键，不需要瞄准

---

## 问题分析

### 问题 1: FLICK 转头没有判定

**根本原因:**
- 在原实现中，FLICK 音符点击后立即设置 `isHit = true`
- 然后调用 `processJudgment()` 清理音符实体
- 0.5 秒后的旋转检测回调执行时，音符实体已经被清理
- 导致旋转检测无法正常工作

**问题代码:**
```java
// 旧代码
noteEntity.setHit(true);  // 立即标记为已击中
// 0.5 秒后检查旋转
PlanetLib.getScheduler().runLater(() -> {
    if (noteEntity.isHit()) {  // 此时实体可能已被清理
        // 旋转检测...
        processJudgment(...);  // 再次清理（但实体已经不存在）
    }
}, 10L);
```

### 问题 2: HOLD 按键检测窗口太短

**根本原因:**
- 原实现使用 100ms 的按键检测窗口
- 当玩家按住键时，事件只在按下时触发一次
- 100ms 后 `isAnyKeyPressed()` 返回 false，即使玩家仍在按住键
- 导致 HOLD 音符无法正确判定

**问题代码:**
```java
// 旧代码
private static final long KEY_PRESS_WINDOW = 100; // 太短！

public boolean isAnyKeyPressed() {
    long currentTime = System.currentTimeMillis();
    return (currentTime - lastKeyPressTime) < KEY_PRESS_WINDOW;
}
```

## 修复方案

### 修复 1: FLICK 音符旋转检测

**修复位置:**
- `NoteEntity.java`: 添加 `waitingForFlickRotation` 字段
- `InputHandler.java`: 重写 `handleFlickNote()` 方法
- `GameSession.java`: 修改 miss 逻辑，跳过等待中的 FLICK 音符

**修复内容:**

#### 1.1 添加等待标记（NoteEntity.java）
```java
// 用于 flick 音符：标记是否正在等待旋转检测
private boolean waitingForFlickRotation = false;
```

#### 1.2 重写 FLICK 处理逻辑（InputHandler.java）
```java
private void handleFlickNote(NoteEntity noteEntity, Player player, JudgmentResult judgment, int timingOffset) {
    if (judgment == JudgmentResult.MISS) {
        processJudgment(noteEntity, JudgmentResult.MISS, timingOffset);
        return;
    }

    // 标记为等待旋转检测
    noteEntity.setWaitingForFlickRotation(true);
    flickWaiting.put(noteEntity, true);

    // 记录点击时的 yaw
    float clickYaw = ViewDirectionHelper.getPlayerYaw(player);

    // 立即播放点击音效（不等旋转结果）
    player.playSound(player.getLocation(), org.bukkit.Sound.ENTITY_PLAYER_HURT, 0.5f, 1.5f);

    // 0.5 秒后检查旋转
    PlanetLib.getScheduler().runLater(() -> {
        if (flickWaiting.getOrDefault(noteEntity, false)) {
            float currentYaw = ViewDirectionHelper.getPlayerYaw(player);
            String targetDirection = noteEntity.getTurn();

            JudgmentResult finalJudgment;
            if (targetDirection != null && ViewDirectionHelper.isRotatingInDirection(
                player, clickYaw, currentYaw, targetDirection)) {
                finalJudgment = JudgmentResult.EXACT;
            } else {
                finalJudgment = JudgmentResult.MISS;
            }

            // 现在才标记为已击中并处理判定
            noteEntity.setHit(true);
            noteEntity.setWaitingForFlickRotation(false);
            gameSession.getScoreManager().recordJudgment(finalJudgment);

            // 显示判定文本（不播放音效，已在点击时播放）
            showJudgmentText(noteEntity, finalJudgment, 0);

            // 清理音符实体
            noteEntity.cleanup();
            entityManager.unregisterEntity(noteEntity.getLinkUUID());

            flickWaiting.remove(noteEntity);
        }
    }, 10L);
}
```

**关键改进:**
1. ✅ 不再立即设置 `isHit = true`
2. ✅ 不再立即调用 `processJudgment()`
3. ✅ 使用 `waitingForFlickRotation` 标记防止被其他逻辑清理
4. ✅ 点击时立即播放音效，提供即时反馈
5. ✅ 0.5 秒后才真正处理判定和清理实体

#### 1.3 保护等待中的 FLICK 音符（GameSession.java）
```java
// 自动移除超过判定线太远的音符
if (distance < -10) {
    // FLICK 音符如果正在等待旋转检测，不要标记为 MISS
    if (!entity.isHit() && !entity.isWaitingForFlickRotation()) {
        scoreManager.recordJudgment(org.cubeRhythm.judgment.JudgmentResult.MISS);
    }
    // 如果是 FLICK 音符正在等待，也不要清理（让 InputHandler 处理）
    if (!entity.isWaitingForFlickRotation()) {
        entityManager.unregisterEntity(entity.getLinkUUID());
    }
}
```

**关键改进:**
1. ✅ 检查 `isWaitingForFlickRotation()` 标记
2. ✅ 等待中的 FLICK 音符不会被标记为 MISS
3. ✅ 等待中的 FLICK 音符不会被提前清理

### 修复 2: HOLD 按键检测窗口

**修复位置:** `InputHandler.java`

**修复内容:**

#### 2.1 延长按键检测窗口
```java
// 旧代码
private static final long KEY_PRESS_WINDOW = 100; // 100ms

// 新代码
private static final long KEY_PRESS_WINDOW = 1000; // 1000ms（1秒）
```

**原理:**
- 将按键检测窗口从 100ms 延长到 1000ms
- 当玩家按住键时，虽然事件只触发一次，但 1 秒内都认为键仍在按下
- 这对于 HOLD 音符的判定足够了，因为：
  - HOLD 音符到达判定线时检查按键状态
  - 如果玩家在 1 秒内按过键，认为正在按住
  - 1 秒的窗口足够覆盖正常的 HOLD 音符判定

**为什么 1 秒合适:**
- HOLD 音符的判定窗口是 ±200ms（JUST）或 ±80ms（EXACT）
- 玩家通常会在音符接近时开始按住键
- 1 秒的窗口足够覆盖从按键到判定的时间
- 不会太长导致误判（玩家松开键后仍被认为在按）

## 修复总结

### 修复的文件

1. **NoteEntity.java**
   - 添加 `waitingForFlickRotation` 字段

2. **InputHandler.java**
   - 延长 `KEY_PRESS_WINDOW` 从 100ms 到 1000ms
   - 添加 `flickWaiting` Map
   - 重写 `handleFlickNote()` 方法
   - 更新 `cleanup()` 方法

3. **GameSession.java**
   - 修改 miss 逻辑，保护等待中的 FLICK 音符

### 修复的问题

| 问题 | 状态 | 说明 |
|------|------|------|
| FLICK 转头没有判定 | ✅ 已修复 | 延迟判定处理，防止实体被提前清理 |
| HOLD 按键检测窗口太短 | ✅ 已修复 | 延长窗口到 1 秒，足够检测持续按键 |

### 技术要点

1. **FLICK 音符的延迟判定**
   - 使用 `waitingForFlickRotation` 标记保护音符
   - 点击时播放音效，提供即时反馈
   - 0.5 秒后才处理判定和清理

2. **HOLD 音符的按键检测**
   - 使用 1 秒窗口近似检测持续按键
   - 不需要瞄准，只检测按键状态
   - 无打击音效，符合规则

3. **防止冲突**
   - FLICK 音符等待期间不会被 miss 逻辑清理
   - HOLD 音符不会被点击判定逻辑处理
   - 各类型音符的判定逻辑互不干扰

## 后续建议

### 1. 调整按键检测窗口

如果 1 秒窗口在实际游玩中不够准确，可以调整：

```java
// 更短的窗口（更严格）
private static final long KEY_PRESS_WINDOW = 500; // 500ms

// 更长的窗口（更宽松）
private static final long KEY_PRESS_WINDOW = 1500; // 1.5秒
```

### 2. 改进按键检测

如果需要更精确的按键检测，可以考虑：
- 使用 Bukkit 的 `PlayerToggleSneakEvent` 等事件追踪按键状态
- 实现按键按下/释放的状态机
- 使用客户端 mod 提供更精确的按键信息

### 3. FLICK 音符反馈优化

可以考虑添加更多反馈：
- 点击时显示"等待转头"提示
- 转头过程中显示角度指示器
- 转头成功时播放额外音效
