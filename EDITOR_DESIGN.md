# 铺面编辑器设计文档

## 概述

本文档描述了 CubeRhythm 铺面编辑器的设计思路。由于在 Minecraft 中实现完整的编辑器存在诸多技术限制，该功能已被标记为废弃。建议使用外部工具进行铺面编辑。

## 设计目标

创建一个游戏内的可视化铺面编辑器，允许玩家：
- 实时预览音符放置位置
- 在 4 个判定面上放置不同类型的音符
- 通过时间轴导航编辑铺面
- 自动保存到 JSON 格式文件

## 核心概念

### 1. 多面坐标系统

游戏使用 4 个判定面围绕玩家：
- **W 面** (前): z = centerZ + 4
- **A 面** (左): x = centerX + 4
- **S 面** (后): z = centerZ - 4
- **D 面** (右): x = centerX - 4

每个面的有效范围：
- 水平范围: ±3 blocks
- 垂直范围: ±3 blocks

### 2. 时间系统

使用 tick-based 时间表示：
- **tick**: 整数，表示当前时间位置
- **stepLength**: 1 拍分成几份（如 4 = 1/4 拍精度）
- **时间计算**: `time = tick / stepLength * (60 / bpm) + preTime`
- **小节计算**: `beat = floor(tick / stepLength) + 1`

### 3. 射线投射系统

用于确定玩家视线与判定面的交点：

```
1. 获取玩家位置和视线方向向量
2. 对每个判定面计算射线与平面的交点
3. 验证交点是否在面的有效范围内
4. 选择最近的有效交点作为目标面
```

**眼睛高度偏移**:
- 正常: +1.6 blocks
- Shift (网格对齐): +1.4 blocks

**网格对齐**: Shift 时坐标对齐到 0.5 block 精度

## 主要组件

### EditorSession (编辑会话)

管理单个玩家的编辑状态：
- 当前时间位置 (tick)
- 选择的音符类型和面
- BPM、预时间、速度等参数
- 已放置的音符列表
- 预览光标实体引用

### EditorManager (会话管理器)

全局单例，管理所有活动的编辑会话：
- 创建/删除会话
- 查询玩家的会话
- 清理所有会话

### EditorPreviewCursor (预览光标)

实时显示将要放置的音符：
- 每 tick 或视角变化时更新
- 使用射线投射计算位置
- 自动检测当前指向的判定面
- 显示半透明的音符预览

### EditorNoteRenderer (音符渲染器)

渲染编辑器中的音符：
- 只渲染可见范围内的音符 (±5 秒)
- 根据当前时间计算音符的 z 偏移
- 支持多面同时渲染

### EditorFaceDetector (面检测器)

自动检测玩家视线指向哪个判定面：
- 对 4 个面进行射线投射
- 计算交点距离
- 验证交点在面的有效范围内
- 返回最近的有效面

## 核心功能

### 1. 进入编辑模式

```
/editor new <chartId>  - 创建新铺面
/editor load <chartId> - 加载现有铺面
```

初始化：
- 创建 EditorSession
- 给予编辑工具（魔杖、音符类型选择器）
- 在所有判定面放置玻璃标记
- 启动预览光标更新任务

### 2. 时间导航

**鼠标滚轮导航**:
- 滚轮向前 (slot 5-7): 前进一格
- 滚轮向后 (slot 1-3): 后退一格

**命令导航**:
- `/b <beat>`: 跳转到指定拍
- `/step <n>`: 设置步长（1/n 拍）

### 3. 音符放置

**右键点击魔杖**:
1. 计算视线与判定面交点
2. 验证坐标在有效范围内
3. 创建音符实体在判定面上
4. 保存音符数据到 JSON

**左键点击魔杖**:
1. 找到预览光标指向的音符
2. 删除音符实体
3. 从 JSON 中移除音符数据

### 4. 音符类型切换

滚动到 slot 1 循环切换：
- TAP → DOUBLE → DRAG → HOLD → FLICK(left) → FLICK(right) → TAP

### 5. 参数设置

```
/editor bpm <value>      - 设置 BPM
/editor pretime <value>  - 设置预时间
/editor speed <value>    - 设置速度
```

### 6. 发光切换

按 F 键切换音符发光状态

## 渲染系统

### 预览光标渲染

**更新频率**: 每 tick (20 TPS) 或视角变化时

**渲染流程**:
1. 删除旧的预览实体
2. 使用射线投射计算交点
3. 检测当前指向的判定面
4. 在交点位置创建 BlockDisplay
5. 设置材质、颜色、发光效果

### 音符渲染

**更新时机**: 时间导航时

**可见范围**: 当前时间 ±5 秒

**渲染流程**:
1. 删除所有旧的音符实体
2. 遍历所有音符
3. 计算音符相对于当前时间的 z 偏移
4. 只渲染可见范围内的音符
5. 在对应判定面上创建 BlockDisplay

**z 偏移计算**:
```
distance = (noteTime - currentTime) * 20 * speed + 4
```

### Action Bar 信息

每 2 ticks 更新一次，显示：
```
BPM: {bpm}   第 {小节号}小节 {拍位置}/{步长}拍 ({时间}s)   共计 {音符数量} Notes
```

## 数据存储

### JSON 格式

直接使用标准的铺面 JSON 格式：

```json
{
  "version": "1.0.0",
  "metadata": {
    "id": "chart_id",
    "title": "Song Name",
    "bpm": 120,
    ...
  },
  "notes": [
    {
      "type": "tap",
      "time": 1.5,
      "face": "w",
      "position": {"x": 0.5, "y": 0.5},
      "glowing": false
    }
  ]
}
```

### 自动保存

- 每次放置/删除音符时自动保存
- 使用 EditorFileUtil 处理 JSON 序列化
- 保存到 `plugins/CubeRhythm/{chartId}.json`

## 技术挑战与限制

### 1. 多面坐标转换复杂

每个判定面有不同的坐标系统，需要复杂的转换逻辑：
- W/S 面: 使用 X-Y 坐标
- A/D 面: 使用 Z-Y 坐标
- 需要根据面旋转和偏移坐标

### 2. 射线投射精度问题

- 需要同时检测 4 个面
- 边界情况处理复杂
- 玩家视角变化频繁，计算开销大

### 3. 实体管理开销

- 预览光标需要每 tick 更新
- 音符渲染需要大量 BlockDisplay 实体
- 频繁创建/删除实体影响性能

### 4. 用户体验限制

- Minecraft 的输入系统不适合精确编辑
- 缺少撤销/重做功能
- 难以实现复杂的选择和批量操作
- 无法显示波形或音频可视化

### 5. 同步问题

- 多个玩家同时编辑同一铺面
- 文件保存冲突
- 实体状态同步

## 替代方案

### 推荐: 外部编辑器

开发独立的桌面应用或 Web 应用：

**优势**:
- 更好的用户界面
- 完整的撤销/重做支持
- 音频波形可视化
- 精确的鼠标/键盘控制
- 批量编辑功能
- 不受 Minecraft 性能限制

**技术栈建议**:
- Web: React + Canvas/WebGL
- 桌面: Electron 或 Tauri
- 音频: Web Audio API 或 JUCE

### 简化方案: 命令行编辑器

如果需要游戏内编辑，可以简化为命令行界面：

```
/note add tap w 0.5 0.5 1.5    - 添加音符
/note delete <id>              - 删除音符
/note list                     - 列出所有音符
/note move <id> <x> <y>        - 移动音符
```

**优势**:
- 实现简单
- 性能开销小
- 适合快速调整

**劣势**:
- 缺少可视化
- 操作繁琐
- 学习曲线陡峭

## 实现状态

当前代码中的编辑器实现已标记为 `@Deprecated`，原因：
1. 技术复杂度过高
2. 性能开销大
3. 用户体验受限
4. 维护成本高

建议使用外部工具进行铺面编辑，游戏内只保留播放功能。
